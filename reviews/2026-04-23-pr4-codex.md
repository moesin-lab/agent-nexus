### Codex 原始输出

**Taste Score**

`7.5/10`。本轮改动的路径选择是克制的：实际 hook 代码与矩阵表基本对齐，仓库根当前也确实只有 `README.md` / `CONTRIBUTING.md` 两个"面向仓库外部读者"的主入口，`scripts/pretool-read-guard:66` 的 `block_kind()` 与 `AGENTS.md:36` 的表述在"拦哪几类、放哪几类"上是自洽的。`docs/product/**`、`docs/ops/**` 现在都在 `docs/**` 这条总规则里，按当前方案不算遗漏。

`README.md` 和 `CONTRIBUTING.md` 现在也还算能归到一类：前者是产品/项目对外介绍；后者虽然复述协作要求，但主要是外部贡献者入口和指针。当前版本里它没有承载独占的内部权威规则，更多是在转述 `AGENTS.md` / `docs/dev/**`。所以 "external" 目前不是错分类，但它成立的前提是：`CONTRIBUTING.md` 继续只做摘要和导航，不能长成唯一真相源。

**Fatal Issues**

- `P1` `external` 这类的"被拦后唯一入口"语义没有闭环，导致文档宣称和实际读取体验不一致。`AGENTS.md:68` 把 `scripts/docs-read --force` 描述成 hook 拦截后的统一入口，`scripts/pretool-read-guard:98` 也把 `README.md` / `CONTRIBUTING.md` 引到这个入口；但 `scripts/docs-read:143` 的 `--force` 只会对 `status=deprecated|placeholder` 或 `adr_status=Superseded|Deprecated` 发 stderr 告警。对 `README.md` / `CONTRIBUTING.md` 这种 `status: active` 的"external"文件，`--force` 实际就是静默 `cat` 全文。结果：hook 阶段说"这是外部导向文档，不是事实源"；真正读取阶段却不再携带任何"external"元信息；`archived` 和 `external` 虽然都走 `--force`，但只有前者有持续告警，后者没有。

### 主 session 响应

#### P1：external 闭环不完整 → **采纳**

属实。原先 `--force` 只对 frontmatter 过时告警，路径型 external 是静默 cat，拦截理由只在 hook stderr 出现一次。修法（commit `97e9d47`）：

- `scripts/docs-read --force` 加一个独立分支：若路径是 `README.md` 或 `CONTRIBUTING.md`（仓库根，含 `$path != */*` 判定避免误匹配 `docs/product/README.md`），stderr 输出"按 --force 读取外部导向文档"告警
- 保持 frontmatter 过时分支原有行为，两类场景用 `if/elif` 互斥
- `AGENTS.md` §docs-read 三模式表里 `--force` 场景描述从"读归档文档"改为"读被 hook 拦截的文档（归档 / 外部导向）"

验证：A/B 两类告警对称；active 文档不触发告警；`docs/product/README.md` 走 B 类（frontmatter placeholder）而不是 A 类 external，不误判。

#### Improvement-1：CONTRIBUTING 边界约束 → **采纳**

`CONTRIBUTING.md` 顶部增补 quote block：
> **边界声明**：本文档只做摘要与外部流程说明；内部协作规则以 `AGENTS.md` 为准，架构与契约以 `docs/dev/**` 为准。CONTRIBUTING 不应长出独占的内部规则——规则需要变化时改 `AGENTS.md` 或 `docs/dev/**`，本文档只在必要时同步指针。

这条声明一方面让读者（人类 / agent）知道边界，另一方面 reviewer 未来看到 CONTRIBUTING 里出现新规则内容时有依据要求迁出。

#### Improvement-2：拦 `docs/product/**` → **拒绝**

采纳 codex 的"先别膨胀"判断。docs/product 当前全部 placeholder，按方案 M 的路径型收窄原则，未来真长出产品话术再单独加路径规则更稳。
