# 子流程 A：对象是外部仓库 / 框架

**何时用**：被评估对象是他人 repo、URL、框架、方法论包。通常需要起 scratch 归档（材料量大）。在主轴步骤 1 之前插入 A.1 + A.2，步骤 1 套用 A.3 的维度模板。

## A.1 Layer 定位（第一段必写）

先回答：**被评估对象跟当前项目是否同一抽象层级？**

常见 layer：

- **产品**：有自己的用户、部署形态、runtime
- **库 / SDK**：供他人嵌入的代码包
- **方法论 / 规范**：纯文档、流程、约定
- **skill 包 / prompt 集合**：面向 AI agent 的指令集

层级错配时（用产品对比 skill 包），逐条比较全是鸡同鸭讲。错配场景下只做"机制点抽取"，不做"架构借鉴"。

## A.2 高效 fetch（不要逐文件读）

1. `gh api repos/<owner>/<repo>/git/trees/<branch>?recursive=1` 一次拿全树，识别目录结构。
2. fetch README 了解自述定位。
3. 挑 2–4 个代表性文件：核心接口定义 / 方法论推导文 / 典型实体样本。
4. 大文件先看 frontmatter / 一级标题 outline，需要再读全文。

目标：用最少 token 覆盖"作者想让别人看到什么 + 实际写成什么样"。

## A.3 推荐维度模板

1. **Layer 定位**（必须是第一段）
2. **可借鉴点**（每点独立一段，带落地建议）
3. **明确不借鉴**（每条必给理由，否则用户会误以为遗漏）
4. **Caveat**（方法论包装、选型偏差、团队规模差异、技术栈差异）
5. **下一步优先级**（按"改动面 × 价值"排序，从最轻量起步）

## 本子流程专属硬规则

- 第一段必须是 Layer 定位；不得跳过。
- "明确不借鉴"清单每条必给理由，不接受只列条目。
- 未拉全 tree 就逐文件 fetch → 返工。
